# IKEv2/IPsec VPN домой

Как заставить телефон автоматически подключаться к домашней сети через роутер
**Keenetic/Netcraze** по LTE/чужому Wi-Fi, видеть домашний LAN и наследовать маршрутизацию
роутера full-tunnel.

## 0. Идея
Телефон = роуминговый клиент домашней сети. Full-tunnel → **весь** трафик телефона
заходит в роутер, а дальше роутер сам применяет свою маршрутизацию/DNS (LAN, выборочный
выход через другой VPN и так далее). На телефоне эту логику настраивать не нужно.

## 1. Пример переменных которые нужно заполнить под свой стенд в файле ios-ikev2-home.mobileconfig
| Токен | Что это | Пример (подставьте своё)     |
|---|---|------------------------------|
| `DDNS_HOST` | Своя A-запись → WAN-IP роутера (резолвится везде, в т.ч. на LTE) | `home.domain.ru`             |
| `KEENDNS_NAME` | KeenDNS-имя роутера = идентичность сертификата сервера | `home.keenetic.pro`          |
| `WAN_IP` | Публичный WAN-IP роутера (нужен публичный IP, не CGNAT) | `203.0.113.10`               |
| `VPN_USER` / `VPN_PASS` | Учётка роутера с доступом к IKEv2 | `vpnuser` / `vpnpass`        |
| `HOME_SSID` | Имя(имена) домашнего Wi-Fi, где VPN должен быть ВЫКЛ | `HomeWiFi_2G`, `HomeWiFi_5G` |
| `POOL` | Пул виртуальных адресов для клиентов | `172.20.8.1…172.20.9.0`      |

**Внимание**: сначала нужно создать A-запись `DDNS_HOST → WAN_IP` и проверить `dig +short DDNS_HOST`.
**Внимание**: нужно у провайдера получить Белый статический IP-адрес, который закреплён за Вашим роутером (не ONT-терминалом).
Публичный адрес должен отображаться не только на сайтах проверки IP, что может говорить о том,
что NAT выполняется на ONT-терминале провайдера. Поэтому нужно перевести порт в режим Bridge
либо выдать публичный IPv4 непосредственно на WAN Вашего роутера без промежуточного NAT.

## 2. Роутер — установка и настройка Keenetic/Netcraze
1. **Компонент:** Настройка системы → *Показать компоненты* → установить
   **IKEv1/IPsec и IKEv2/IPsec VPN-серверы** (имя в CLI — `virtual-ip-server`).
   Ставится новый образ прошивки и роутер **перезагружается** (~1–3 мин, конфиг сохраняется).
2. **KeenDNS:** Доменное имя → *Режим работы - прямой доступ* → *Не использовать IPv6 - ставим галку*
   → *Имя Вашего устройства Keenetic/Netcraze - вводим, например, home* → *Зарегистрировать*
   → Появится доменное имя, например, *home.keenetic/netcraze.pro/link или какой Вы выбрали*.
3. **Приложения → VPN-сервер IKEv2/IPsec:** Включить ползунок, Тапаем по надписи *VPN-сервер IKEv2/IPsec*
   → *Начальный IP-адрес - можно оставить по умолчанию* → *Пул IP-адресов - можно оставить по умолчанию*
   → *Доступ к сети - выбираем Домашняя сеть* → *Режим оптимизации - По умолчанию*
   → *Множественный вход - включаем если хотим несколько устройств под одной учёткой VPN*
   → *NAT для клиентов - включаем обязательно* (нужно для full-tunnel и интернета) → дать доступ учётке `VPN_USER`.
   Это автоматически выпустит сертификат **Let's Encrypt** на `KEENDNS_NAME` и создаст
   crypto-конфиг (traffic-selectors `0.0.0.0/0` = предлагается full-tunnel). Учётке admin доступ лучше не давать!
4. **Порты WAN:** разрешить **UDP 500 + 4500** (открываются сами).
5. **Проверка (SSH):** `Приложения → *VPN-сервер IKEv2/IPsec* → *Статистика подключений - покажет список подключенных
   клиентов*.

## 3. Зачем KeenDNS, если используем свою A-запись?
У них **разные роли**:
- `DDNS_HOST` (Ваш домен) = **адрес, куда ходит телефон**. Он обязан резолвиться у
  сотового оператора — поэтому берём свой домен (операторы отдают SERVFAIL на `keenetic.link` и другие).
- `KEENDNS_NAME` = **идентичность сертификата сервера**. IKEv2-сервер Keenetic/Netcraze умеет
  получить и предъявить сертификат только через KeenDNS (авто Let's Encrypt), подсунуть
  ему сертификат на свой домен штатно нельзя. Этим сертификатом сервер доказывает, кто он.

В IKEv2 эти два значения — **разные поля**, поэтому: телефон ходит на `DDNS_HOST`
(резолвится) и сверяет сертификат с `KEENDNS_NAME` (**сравнение строки — без DNS-запроса**).
KeenDNS здесь — просто `фабрика сертификата`, телефон его никогда не резолвит.

## 4. Профиль iOS — построчно (`.mobileconfig`)
Укажу только значимые строки, остальное — обязательная `обвязка` plist.
```
PayloadIdentifier (внешний) ru.<Ваш_домен>.vpn.home  # уникальный id профиля, переустановка с ТЕМ ЖЕ id
                                                     # ЗАМЕНЯЕТ профиль на месте (удалять не нужно)
PayloadUUID       (внешний)  <uuid>                  # держать неизменным между правками — тогда iOS обновляет, а не плодит копии
VPNType           IKEv2                              # нативный IKEv2-клиент iOS
RemoteAddress     DDNS_HOST                          # КУДА ходить — телефон резолвит через DNS (должно резолвиться на LTE)
RemoteIdentifier  KEENDNS_NAME                       # КТО должен быть на том конце = идентичность сертификата.
                                                     # ТОЛЬКО сравнение строки, не резолвится. ← ошибка:
                                                     # должно ТОЧНО совпадать с leftid/CN сертификата сервера
                                                     # (иначе в логе strongSwan no matching peer config found)
LocalIdentifier   VPN_USER                           # IKE-идентичность = имя пользователя
AuthenticationMethod None                            # не требуется аутентификация сертификатом...
ExtendedAuthEnabled 1                                # ...а EAP (логин/пароль)
AuthName          VPN_USER                           # логин EAP
AuthPassword      VPN_PASS                           # пароль EAP (если удалить строку — спросит один раз при подключении)
ServerCertificateIssuerCommonName ""                 # пусто = доверие через системное хранилище CA (LE доверенный → импорт не нужен)
EnablePFS         0                                  # PFS-группа не задана
IKE/Child SA параметры AES-256 / SHA2-256 / DH14     # должны попадать в набор, поддерживаемый сервером
OnDemandEnabled   1                                  # включить логику авто-подключения (смотрите раздел 5)
OnDemandRules[]   (по порядку, срабатывает первое совпадение):
   1 Disconnect  WiFi + SSID из {HOME_SSID}          # дома → VPN ВЫКЛ (Вы и так в домашней сети (LAN))
   2 Connect     WiFi (любой другой SSID)            # чужой Wi-Fi → подключиться
   3 Connect     Cellular                            # LTE → подключиться
   4 Connect     (fallback)                          # всё прочее → подключиться
```

## 5. Настройка iPhone и авто-подключение по SSID
Установка: AirDrop/почта/мессенджер с `.mobileconfig` → тапаем по файлу → Настройки → *Профиль загружен* → Установить.
Проверка: Настройки → VPN → профиль **выбран** как активный, ⓘ → *Подключение по требованию/запросу* = ВКЛ.
Почему работает это работать будет само: правила On-Demand/Подключение по требованию/запросу живут в сетевом слое iOS `neagent`,
который **пересматривает их при каждой смене сети** и сверяет **текущий SSID Wi-Fi**. Уход из
`HOME_SSID` или переход на `LTE` вызывает `Connect` автоматически, возврат на `HOME_SSID` —
`Disconnect`. Дома Cтатус VPN `Не подключено` — это **правильно**. Не выключайте вручную
переключатель `Статус VPN` — это приостанавливает On-Demand/Подлкючение по требованию/запросу.

## 6. Android — strongSwan + Tasker
Качаем в GooglePlay **strongSwan** (не встроенный VPN): только он позволяет задать идентичность сервера
отдельно от адреса подключения (та же идея, что `RemoteIdentifier` на iOS).
1. strongSwan → **Добавить VPN профиль**:
    - Server: `DDNS_HOST`
    - VPN Тип: **IKEv2 EAP (Логин/Пароль)**
    - Логин: `VPN_USER`, Пароль: `VPN_PASS`
    - CA certificate: **Выбрать автоматически** (LE доверенный, импорт не нужен)
    - *Show advanced* → **Server identity: `KEENDNS_NAME`** ← ключевое поле = `RemoteIdentifier`
2. Сохранить → тапаем по профилю для подключения (full-tunnel включается сам, сервер отдаёт `0.0.0.0/0`).
3. Авто-подключение — **Tasker + AutoInput**:
    - Условие/триггер: состояние *Wi-Fi подключён, SSID ≠ HOME_SSID* **ИЛИ** *включены мобильные данные*.
    - Действие: у strongSwan нет команды запуска, поэтому через **AutoInput** открыть strongSwan и
      нажать на профиль, в задаче выхода — нажать *Disconnect* при возврате на `HOME_SSID`.
    - Рекомендую создать **ярлык профиля на рабочий стол** → один тап, когда уходите из дома.

## 7. На Android нет нативного авто-подключения по SSID
В Android **нет триггера VPN по SSID** — есть только *Always-on VPN* (всегда включён, в том числе дома).
А strongSwan **не предоставляет intent/API для запуска** осознанно, поэтому запустить его
программно к сожалению нечем. Получается что автоматизация вынуждена **эмулировать нажатия** (Tasker+AutoInput),
что ненадёжно и может конечно ломаться после обновлений приложения/ОС. На iOS это есть нативно
(On-Demand/Подключение по требованию/запросу), на Android такой фишечки — нет. Кому свежих яблочек 🍏? 😁

## 8. Быстрая проверка и грабли на которые попал
- Подключился? Роутер `show ipsec` → `1 up`, у клиента адрес из `POOL`, `0.0.0.0/0 === <ip>/32` или через веб-морду
  *Статистика подключений*.
- Full-tunnel? Проверяем какой у IP на телефоне = должен быть IP роутера/дома, а не оператора.
- **Не подключается + в логе роутера `no matching peer config found`** → `RemoteIdentifier` /
  *Server identity* ≠ имя из сертификата сервера. Исправить точно на `KEENDNS_NAME`.
  (Где взять истину: Подключаемся к роутеру по SSH `show ipsec` → `local: [<имя>] ... cert: "CN=<имя>"`.) или через
  веб-морду *Доменное имя*
- Аутентификация падает после успешного шифрования → неверный `VPN_PASS` или у учётки нет доступа к IKEv2.
- Эндпоинт должен оставаться доступным: если WAN-IP динамический, держите `DDNS_HOST` авто-обновляемым.
